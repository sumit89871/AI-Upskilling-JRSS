# LLM Fundamentals Overview

## 1. What an LLM is

An LLM is a Large Language Model.

Simple meaning:

```text
An LLM is a model that takes text input and generates text output.
```

More practical meaning:

An LLM looks at the text you give it, breaks that text into smaller pieces called tokens, and predicts what tokens should come next.

It can be used for:

- summarization
- question answering
- extraction
- classification
- test case generation
- code assistance
- requirement analysis
- RAG answers
- agent reasoning

Important:

An LLM does not automatically know your private project documents, company rules, database data, or latest internal updates. Your application must provide that information through the prompt, RAG, tools, or APIs.

## 2. The most important beginner idea

An LLM is not the full application.

The model generates output. The application around the model handles:

- user input
- prompt building
- API call
- API key handling
- RAG retrieval
- tool calling
- validation
- security
- logging
- UI
- FastAPI response
- error handling

Beginner memory:

```text
LLM = text generation engine
AI application = software system that uses the model safely
```

## 3. Beginner mental model

Use this mental model:

```text
Prompt text -> tokens -> model predicts next tokens -> response text
```

Expanded:

```text
User writes question
Application builds prompt
Tokenizer converts text into tokens
Model predicts likely next tokens
Generated tokens are converted back to readable text
Application validates and returns the response
```

This is the core flow behind hosted APIs, local models, chatbots, RAG systems, and agents.

## 4. Model vs application

### Model

The model is the trained system that generates text.

Example:

```text
GPT model, Gemini model, Llama model, Mistral model
```

### Application

The application is the software you build around the model.

Example:

```text
AI QA assistant, chatbot UI, FastAPI backend, RAG search app
```

### Comparison

| Concept | What it does |
|---|---|
| Model | Generates or analyzes text |
| Application | Sends input, handles context, validates output, calls tools, shows result |

Common beginner mistake:

Saying "I built an LLM" when you actually built an application that calls an LLM API.

Better interview wording:

```text
I built an AI application that uses an LLM for generation and uses RAG for grounding.
```

## 5. LLM vs chatbot app

An LLM is the model.

A chatbot app is a user-facing application that uses a model.

Example:

```text
LLM = engine
Chatbot app = car dashboard, steering, safety controls, and user interface around the engine
```

A chatbot app may include:

- login
- chat history
- prompt templates
- model API calls
- RAG retrieval
- safety checks
- file upload
- UI
- monitoring

The LLM alone does not provide all of that.

## 6. Prompt vs completion

### Prompt

The prompt is the input given to the model.

Example:

```text
Generate three test cases for password reset.
```

### Completion / response

The completion or response is the generated output.

Example:

```text
1. Valid email receives reset link.
2. Unknown email receives a safe generic message.
3. Expired reset link is rejected.
```

### Comparison

| Concept | Meaning |
|---|---|
| Prompt | Input instruction or question |
| Completion/response | Output generated by the model |

In chat APIs, people often say "response" instead of "completion."

## 7. Messages format

Many chat APIs use a list of message dictionaries.

Example:

```python
messages = [
    {"role": "system", "content": "You are a helpful QA assistant."},
    {"role": "user", "content": "Generate test cases for login."},
]
```

Syntax breakdown:

- `messages` is a Python variable.
- `=` assigns a value.
- `[]` creates a list.
- each `{...}` is a dictionary.
- `"role"` tells the model who is speaking.
- `"system"` usually gives behavior instructions.
- `"user"` contains the user's request.
- `"content"` stores the actual text.
- commas separate list items and dictionary fields.

What the developer creates manually:

- system instruction text
- user request text
- message structure

What the model provider handles:

- tokenization
- model inference
- response generation

## 8. Inference

Inference means running the model to get an output.

Simple meaning:

```text
inference = using the trained model
```

Training is when a model learns from large data.

Inference is when you send a prompt and get a response.

In AI Engineer work, you usually do inference:

```text
prompt -> model API -> response
```

You usually do not train large foundation models from scratch.

## 9. Attention basics

Attention is a mechanism that helps the model focus on relevant parts of the input while generating output.

Beginner meaning:

```text
attention helps the model decide which earlier tokens matter for the next token
```

Example:

Prompt:

```text
The user forgot her password. Generate test cases for password reset.
```

When generating test cases, the model should pay attention to:

- `forgot`
- `password`
- `test cases`
- `password reset`

You do not need to calculate attention manually. You only need to understand that modern LLMs use attention to connect related parts of text.

## 10. Transformer basics

A transformer is the neural network architecture behind many modern LLMs.

Simple meaning:

```text
Transformer = model architecture that uses attention to process text effectively
```

Why it matters:

Transformers made it practical to train large models that can handle long text patterns, instructions, code, reasoning-like tasks, and multilingual text.

For implementation, you do not need transformer math. You need to know:

- many LLMs are transformer-based
- they process tokens
- attention is a key idea
- they generate output by predicting tokens

## 11. Foundation model

A foundation model is a large model trained broadly so it can be adapted to many tasks.

Examples of tasks:

- summarization
- code generation
- chat
- extraction
- classification
- reasoning support

Beginner meaning:

```text
foundation model = general-purpose base model used for many applications
```

Your AI QA assistant may use a foundation model through an API instead of training a new model from scratch.

## 12. MoE basics

MoE means Mixture of Experts.

Simple meaning:

```text
MoE = model design where different expert parts handle different kinds of inputs
```

Beginner analogy:

A company may have specialists for legal, testing, finance, and coding. A supervisor routes work to the right specialist.

In an MoE model, not every part of the model is used for every token. The system routes work to selected experts.

Why beginners should know this:

You may hear model descriptions mention MoE. You do not need to implement MoE for this course, but you should know it is an architecture approach used to scale models efficiently.

## 13. Hallucination

Hallucination means the model produces an answer that sounds confident but is wrong, unsupported, or invented.

Example:

User:

```text
What is our company's refund policy?
```

Bad model answer:

```text
Refunds are always processed within 3 days.
```

Problem:

The model may have invented this if the policy was not provided.

## 14. Hallucination vs wrong business logic

These are related but not identical.

| Concept | Meaning |
|---|---|
| Hallucination | Model invents unsupported information |
| Wrong business logic | Application rules or code are incorrect |

Example hallucination:

The model invents a policy that was never in the documents.

Example wrong business logic:

Your code routes all high-priority tickets to the wrong team because the routing condition is incorrect.

Both are bad, but fixes are different.

- hallucination fix: grounding, RAG, better prompts, citations, validation
- business logic fix: correct code, tests, requirements review

## 15. Grounding

Grounding means giving the model reliable context so its answer is based on provided information.

Simple meaning:

```text
grounding = answer using trusted context, not only model memory
```

Example:

Instead of asking:

```text
What is the refund policy?
```

you provide:

```text
Use this policy text:
Refunds are processed within 7 business days...

Question:
What is the refund policy?
```

Grounding reduces hallucination risk, but it does not remove all risk. You still need review and validation.

## 16. RAG mental model

RAG means Retrieval Augmented Generation.

Use this mental model:

```text
Question -> retrieve context -> add context to prompt -> model answers using context
```

RAG is useful when the answer should come from documents, notes, policies, manuals, or project knowledge.

RAG does not permanently train the model. It provides relevant context at answer time.

## 17. Common beginner mistakes

- Thinking an LLM is the entire application.
- Expecting the model to know private company data.
- Treating every confident answer as true.
- Using no grounding for knowledge-based questions.
- Confusing hallucination with a normal code bug.
- Thinking bigger models remove the need for validation.
- Thinking RAG permanently teaches the model.
- Using model output directly without checking format or sources.
