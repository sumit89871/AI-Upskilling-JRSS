# LLM Interview Questions

Use these answers for Mettl-style screening, POC discussion, and AI Engineer interviews.

## 1. What is an LLM?

Short answer:

An LLM is a Large Language Model that takes text input and generates text output.

Expanded answer:

An LLM is trained on large amounts of text and learns patterns in language. At inference time, it receives a prompt, processes tokens, predicts likely next tokens, and returns a response.

Project example:

In the final QA assistant POC, an LLM can generate test scenarios from a requirement.

Common wrong answer:

"An LLM is the full chatbot application."

Why this is wrong:

The LLM is only the model. The application handles prompts, APIs, RAG, tools, validation, and UI.

## 2. What is the difference between a model and an application?

Short answer:

The model generates output. The application manages how the model is used.

Expanded answer:

The application handles user input, prompt templates, API calls, RAG context, tool calling, validation, logging, security, and response formatting.

Project example:

A FastAPI service that calls Gemini is an application using a model, not the model itself.

Common wrong answer:

"If I call OpenAI API, I built a model."

Why this is wrong:

You integrated a hosted model into your application.

## 3. What is a prompt?

Short answer:

A prompt is the input instruction or question sent to the model.

Expanded answer:

The prompt tells the model what task to perform and may include output format, context, examples, or constraints.

Project example:

```text
Generate positive, negative, and edge test cases for this login requirement.
```

Common wrong answer:

"Prompt is the model output."

Why this is wrong:

Prompt is input. Completion or response is output.

## 4. What is a completion or response?

Short answer:

It is the text generated by the model after receiving a prompt.

Expanded answer:

In older APIs, generated output is often called completion. In chat APIs, it is commonly called response or assistant message.

Project example:

The generated list of login test cases is the model response.

Common wrong answer:

"Completion means the task is production-ready."

Why this is wrong:

Completion only means generated output. It still needs validation and review.

## 5. What is a token?

Short answer:

A token is a small unit of text processed by the model.

Expanded answer:

A token may be a word, part of a word, punctuation, or spacing. Models process and limit text using tokens, not exact human words.

Project example:

Token limits affect how much RAG context can fit into a prompt.

Common wrong answer:

"One token is always one word."

Why this is wrong:

Some words are split into multiple tokens, and punctuation can be separate tokens.

## 6. What is tokenization?

Short answer:

Tokenization is the process of splitting text into tokens.

Expanded answer:

Before a model processes text, the tokenizer converts it into token IDs. The model uses those token IDs internally.

Project example:

When a long requirement document is too large, tokenization and token limits influence how you chunk it for RAG.

Common wrong answer:

"Tokenization means summarizing text."

Why this is wrong:

Tokenization splits text into model-readable pieces. It does not summarize meaning.

## 7. What is context window?

Short answer:

Context window is the maximum token space a model can consider in one request.

Expanded answer:

It includes system instructions, user prompt, chat history, retrieved context, tool results, and generated output.

Project example:

In a RAG system, you cannot pass every document chunk. You retrieve only relevant chunks that fit within the context window.

Common wrong answer:

"Context window is only the user question."

Why this is wrong:

It includes all text sent to and generated by the model in that request.

## 8. What are embeddings?

Short answer:

Embeddings are numeric vectors that represent text meaning.

Expanded answer:

Embeddings let applications compare text by semantic similarity. They are commonly used in vector databases for RAG retrieval.

Project example:

The final POC can embed requirement notes and search for chunks relevant to a user question.

Common wrong answer:

"Embedding is the answer generated by the LLM."

Why this is wrong:

Embedding is used for search and similarity, not as the final user answer.

## 9. What is the difference between embedding and token?

Short answer:

A token is a text unit used for model processing. An embedding is a numeric vector representing meaning.

Expanded answer:

Tokens are used in generation and context limits. Embeddings are used in semantic search and RAG retrieval.

Project example:

The LLM prompt uses tokens. The vector database stores embeddings.

Common wrong answer:

"Tokens and embeddings are the same because both represent text."

Why this is wrong:

They are used for different purposes in the system.

## 10. What is attention?

Short answer:

Attention helps the model focus on relevant parts of the input while generating output.

Expanded answer:

Modern transformer models use attention to relate tokens to other tokens. This helps the model use earlier parts of the prompt when predicting the next token.

Project example:

When generating test cases, the model should pay attention to the requirement, constraints, and requested output format.

Common wrong answer:

"Attention means the model understands like a human."

Why this is wrong:

Attention is a model mechanism, not human understanding.

## 11. What is a transformer?

Short answer:

A transformer is a neural network architecture used by many modern LLMs.

Expanded answer:

Transformers use attention to process token relationships effectively. You do not need transformer math for basic AI Engineer work, but you should know many LLMs are transformer-based.

Project example:

OpenAI, Gemini, and many local models are built using transformer-style architectures or related designs.

Common wrong answer:

"Transformer is an API endpoint."

Why this is wrong:

Transformer is a model architecture, not an API route.

## 12. What is temperature?

Short answer:

Temperature controls randomness in model output.

Expanded answer:

Lower temperature produces more consistent output. Higher temperature gives more variation. Use low temperature for JSON, classification, extraction, and strict test case formats.

Project example:

For generating structured API test JSON, use low temperature.

Common wrong answer:

"Higher temperature means better intelligence."

Why this is wrong:

Higher temperature means more randomness, not more correctness.

## 13. What is top_p?

Short answer:

`top_p` controls how wide the model's token choice pool is during generation.

Expanded answer:

It is a sampling parameter. At beginner level, avoid changing both `temperature` and `top_p` randomly. Start simple and test output quality.

Project example:

For predictable POC output, keep sampling settings controlled.

Common wrong answer:

"top_p controls input length."

Why this is wrong:

Input length is related to context window. `top_p` affects generation sampling.

## 14. What is max tokens?

Short answer:

`max_tokens` usually limits how many output tokens the model can generate.

Expanded answer:

If it is too low, the answer may be cut off. If it is too high, the response may be longer, slower, and more expensive.

Project example:

Set enough `max_tokens` for a test case JSON response so the model does not stop halfway.

Common wrong answer:

"max_tokens means maximum words."

Why this is wrong:

It means tokens, not words.

## 15. What is hallucination?

Short answer:

Hallucination is when the model generates confident but wrong or unsupported information.

Expanded answer:

This often happens when the model lacks reliable context or is asked about facts it does not know. Hallucination is a design risk in AI applications.

Project example:

If the model invents a password policy not found in the requirement documents, that is hallucination.

Common wrong answer:

"Hallucination is only a UI bug."

Why this is wrong:

Hallucination is a model output reliability issue, not just a frontend problem.

## 16. What is grounding?

Short answer:

Grounding means giving the model trusted context so it answers based on that context.

Expanded answer:

Grounding reduces hallucination risk by anchoring the answer to documents, retrieved chunks, tool results, or approved data.

Project example:

A RAG QA assistant retrieves requirement notes and includes them in the prompt before answering.

Common wrong answer:

"Grounding guarantees the answer is always correct."

Why this is wrong:

Grounding reduces risk but still needs good retrieval, prompts, validation, and review.

## 17. What is RAG?

Short answer:

RAG means Retrieval Augmented Generation. It retrieves relevant context before generating an answer.

Expanded answer:

RAG is useful when answers should come from documents or knowledge bases. The app retrieves relevant chunks, adds them to the prompt, and asks the model to answer using that context.

Project example:

The final POC can retrieve requirement notes before generating test cases.

Common wrong answer:

"RAG permanently trains the model."

Why this is wrong:

RAG gives context at answer time. It does not update model weights.

## 18. RAG vs fine-tuning?

Short answer:

RAG provides external context at answer time. Fine-tuning changes model behavior through additional training.

Expanded answer:

Use RAG for changing documents, policies, project notes, and grounded answers. Fine-tuning may help with repeated style or task behavior, but it is not usually the first choice for private knowledge.

Project example:

Use RAG for a QA knowledge assistant over local requirement files.

Common wrong answer:

"Fine-tuning is always better than RAG."

Why this is wrong:

For knowledge that changes often, RAG is usually more practical.

## 19. Hosted model vs local model?

Short answer:

A hosted model runs on provider infrastructure through an API. A local model runs on your machine or local server.

Expanded answer:

Hosted models are easier to start with and often stronger, but require API keys, internet, cost control, and data review. Local models offer more local control but may be slower or weaker on normal laptops.

Project example:

Use mock mode first, optional OpenAI/Gemini hosted mode later, and optional Ollama local model for local experimentation.

Common wrong answer:

"Local models are always better because they are private."

Why this is incomplete:

Local models can improve control, but privacy also depends on the full application design and data flow.

## 20. Open model vs closed model?

Short answer:

An open model is more accessible for local use or adaptation depending on license. A closed model is usually provider-controlled and accessed through an API.

Expanded answer:

Open models can give more control, but you must check licensing. Closed models are often strong and easy to use through APIs, but you have less visibility and more provider dependency.

Project example:

For a local demo, you may use an open model through Ollama. For better quality, you may call a closed hosted model API.

Common wrong answer:

"Open means free for any use."

Why this is wrong:

Open model licenses vary. Enterprise use must check license terms.
