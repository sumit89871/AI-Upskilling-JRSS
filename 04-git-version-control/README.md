# 04 Git Version Control

## 1. What This Module Is

This module teaches Git from complete beginner level.

Git is a version control tool. Version control means your project can keep a history of changes instead of becoming one folder full of random final, final2, and final_latest files.

In this course, you will use Git to track:

- Python practice files
- FastAPI backend changes
- RAG experiment changes
- Docker and Kubernetes configuration changes
- final POC progress

## 2. Why It Matters

AI Engineer work changes quickly. You may try one prompt, change a retrieval function, update a Dockerfile, or break a FastAPI endpoint while experimenting.

Git helps because it lets you:

- save checkpoints
- compare older and newer versions
- see what changed
- work on branches
- collaborate through GitHub or another remote platform
- avoid losing working code
- explain project evolution in an interview

## 3. What The Learner Should Finish Knowing

After this module, you should be able to explain and use:

- Git vs GitHub
- repository
- working directory
- staging area
- commit history
- `.git` folder
- tracked and untracked files
- staged and unstaged changes
- `.gitignore`
- branch
- merge
- clone
- pull
- push
- simple conflict resolution
- feature branch workflow

## 6. Practical Scope

This module focuses on Git commands you actually need for labs and POCs.

You will not go deep into advanced history rewriting, rebasing, cherry-picking, or complex enterprise Git policies yet.

## 7. What Not To Over-Focus On

Do not start with advanced Git.

First become strong with:

- `git status`
- `git add`
- `git commit`
- `git log`
- `.gitignore`
- `git branch`
- `git switch`
- `git clone`
- `git pull`
- `git push`

If these are clear, advanced Git becomes easier later.

## 8. How This Helps In AI Engineer JRSS / Mettl / POC / Interview

- **AI Engineer JRSS labs**: you can keep lab work organized and recover after mistakes.
- **Mettl screening**: Git command questions often test `add`, `commit`, `push`, branch, and conflict concepts.
- **POC demo**: reviewers may expect clean project structure, `.gitignore`, and no committed secrets.
- **Interview**: you can explain how you tracked changes while building FastAPI, RAG, Docker, and final POC features.
